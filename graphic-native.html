BXD AI Native · 그래픽

Graphic을
AI 네이티브화한다

되는 것과 안 되는 것이 있다. 전부 다 하려 하지 않는다.
대표 케이스를 먼저 선행하고, 유형별로 구조화하고, 규칙 하에 명명하고 태그화한다.

원칙 01
양이 아니라
대표 케이스 선행
전체를 한번에 정리하지 않는다. 대표 이미지·영상 1개씩 먼저. 그게 나머지의 기준이 된다.
원칙 02
유형별 구조화
규칙 하에 명명
콘텐츠가 무엇인지 명명하고 분류한다. 이름이 없으면 찾을 수 없고, AI도 분류할 수 없다.
원칙 03
가능한 것은
코드로 변환
SVG·JSON으로 변환 가능한 것은 변환한다. 안 되는 것은 색상·레이아웃을 텍스트 개요로.
원칙 04
파일 아닌
링크로 호출
GitHub에 올리고 URL을 배포한다. 파일을 주고받는 게 아니라 링크 하나가 항상 최신을 가리킨다.
1
되는 것 vs 안 되는 것 — 먼저 구분한다
모든 그래픽을 코드화할 수는 없다
✓ 코드화 가능 — SVG / JSON으로 변환
로고 · 심볼 · 아이콘
벡터 기반. Figma에서 SVG로 직접 내보내기 가능. 코드로 읽히고 AI가 직접 참조.
단순 기하 키비주얼 · 패턴
도형·선·그라디언트로 구성된 키비주얼. 구조가 단순하면 SVG로 변환 가능. 색상·레이아웃 정보가 코드 안에 보존됨.
컬러 팔레트 · 타이포 스펙
수치로 정의 가능한 것들. JSON 토큰으로 변환하면 AI가 값으로 읽고 적용.
레이아웃 그리드 · 여백 규칙
수치와 비율로 정의 가능. MD나 JSON으로 구조화하면 AI가 레이아웃 기준으로 참조.
✕ 코드화 불가 — 대안 처리 필요
사진 · 인물 이미지 · 행사 현장
래스터 이미지. SVG 변환 불가. 코드로 구조를 표현할 수 없음.
영상 · 모션 그래픽
파일 자체를 코드화할 수 없음. 영상 URL + 메타데이터로 관리.
복잡한 일러스트 · 합성 이미지
레이어·효과가 복잡해 SVG 변환 시 깨지거나 의미 없는 코드가 됨.
3D 렌더 · 특수 효과 그래픽
벡터로 표현 불가. 이미지 파일로만 존재 가능.
코드화 안 되는 것들의 대안 처리
변환이 안 되는 그래픽은 색상·레이아웃·무드를 텍스트 개요로 작성하고 분류한다. "이 이미지는 BXD 그린 계열, 미니멀, 테크 톤, 행사용" — 이렇게 태그와 설명을 붙이면 AI가 이미지 파일은 못 읽어도 어떤 맥락의 자산인지는 이해할 수 있다. 파일 URL + 메타데이터 MD가 이 그룹의 아카이빙 방식.
2
콘텐츠 유형 명명 & 분류
대표 영상·이미지를 정하는 게 먼저
유형명 정의 & 대표 예시 명명 규칙 변환 포맷
Brand Film 브랜드 전체를 대표하는 영상. 연 1회 수준. 가장 공식적인 영상 자산. brand-film-YYYY.mp4 URL+MD mp4 + MD
Key Visual 캠페인·시즌 대표 이미지. 온오프라인 공통 기준 비주얼. kv-{campaign}-{YYYY}.svg
kv-{campaign}-{YYYY}.jpg		 SVG 가능 SVG / JPG
Event Visual 특정 행사·이벤트용 비주얼. 날짜·장소 정보 포함. ev-{event-name}-{YYYYMM}.svg SVG 가능 SVG
Tech Content 기술·제품 소개 콘텐츠. 블로그·영상·인포그래픽 등. tech-{topic}-{type}-{YYYY}.ext URL+MD JPG/MP4 + MD
Marketing Asset SNS·배너·광고 등 마케팅 채널용 산출물. mkt-{channel}-{campaign}-{YYYYMM}.ext URL+MD JPG/PNG + MD
Pattern / Texture 브랜드 패턴·텍스처. 반복 사용 배경 요소. pattern-{name}-{variant}.svg SVG 가능 SVG
Goods Visual 굿즈 시안 및 최종 결과물 이미지. goods-{item}-{YYYYMM}.jpg 이미지만 JPG + MD
대표 케이스를 먼저 정한다. 전체 아카이브를 한번에 정리하려 하지 않는다. 유형별로 대표 1개를 먼저 골라 명명 규칙을 검증하고, 그게 나머지의 기준이 된다. Brand Film 1개, Key Visual 1개, Event Visual 1개 — 이 3개로 시작한다.
3
명명 규칙 — 이름이 곧 분류다
파일명만 봐도 무엇인지 알 수 있어야 한다
# 명명 구조
{유형}-{내용/캠페인명}-{날짜}.{확장자}

ev-ncloud-day-20240910.svg ← Event Visual, NCLOUD DAY, 2024.09.10
mkt-instagram-ai-launch-202405.jpg ← 마케팅, 인스타, AI 런칭 캠페인
tech-hyperclova-thumbnail-2024.jpg ← 테크 콘텐츠, HyperCLOVA, 썸네일
brand-film-2024.mp4 ← 브랜드 필름, 2024년
pattern-grid-green-light.svg ← 패턴, 그린 그리드, 라이트 버전
규칙 3가지만 지키면 된다. ① 소문자만 사용, 공백 없이 하이픈(-) 연결. ② 날짜는 YYYYMM 또는 YYYYMMDD — 일관되게. ③ 유형 코드(kv / ev / mkt / tech / brand-film / pattern / goods)를 항상 앞에. 이 규칙이 폴더 구조, URL, 태그 체계와 모두 연결된다.
4
태그 체계 — 찾고 쓰기 위한 구조
파일명 + 태그가 같이 있어야 AI가 맥락을 이해한다
유형 태그 — 무엇인가
key-visual event-visual brand-film tech-content marketing-asset pattern goods-visual
채널 태그 — 어디서 쓰나
online offline social instagram youtube display-ad print signage presentation
무드 태그 — 어떤 느낌인가
tech minimal bold warm dark light motion static
상태 태그 — 현재 상태
representative archived in-use deprecated pilot

태그는 브랜드 활용 기준이 된다. "테크 콘텐츠 썸네일로 쓸 수 있는 미니멀한 이미지 찾아줘" — 태그가 잘 붙어 있으면 AI가 맥락에 맞는 자산을 바로 찾아준다. 태그 설계는 검색을 위한 것이기도 하지만, AI에게 자산의 맥락을 설명하는 언어이기도 하다.
5
먼저 할 대표 케이스 — 유형별 1개씩
원칙 01 — 양이 아니라 대표 케이스 선행
SVG 변환 대표 Key Visual 1개
가장 최근 + 가장 많이 쓰인 키비주얼 선정
Figma에서 SVG로 내보내기
불필요 메타데이터 정리
명명 규칙 적용 후 GitHub 업로드
MD 메타데이터 파일 작성 (태그 포함)
URL 확인 및 Foundation 파일에 등록
→ 이게 Key Visual 정규화의 기준 케이스
URL+MD 대표 Brand Film 1개
가장 최근 브랜드 대표 영상 선정
영상 파일 또는 유튜브 URL 확정
명명 규칙 적용 (brand-film-YYYY)
MD 메타데이터 파일 작성 (무드·용도·태그)
색상·레이아웃·톤을 텍스트로 개요 작성
아카이빙 사이트에 링크+메타데이터로 등록
→ 이게 영상 자산 아카이빙의 기준 케이스
SVG 변환 대표 Event Visual 1개
가장 반복 빈도 높은 행사 비주얼 선정
SVG 변환 가능 여부 먼저 확인
가능하면 SVG, 불가하면 URL+MD 방식으로
행사 유형 태그 체계 검증
명명 규칙 적용 후 업로드
Before·During·After 구조로 활용 기준 작성
→ 행사 비주얼 정규화의 기준 케이스
URL+MD 대표 Tech Content 1개
가장 많이 참조된 테크 콘텐츠 선정
썸네일 이미지 명명 규칙 적용
콘텐츠 주제·무드·채널 태그 작성
URL + MD 메타데이터로 아카이빙
Brand Prompt Library 소스로 연결
AI가 이 케이스를 참조해 유사 콘텐츠 제안 가능
→ 테크 콘텐츠 아카이빙의 기준 케이스
4개 대표 케이스가 완성되면 나머지는 같은 방식으로. 명명 규칙·태그 체계·MD 구조가 이 4개를 통해 검증된다. 이후 팀원 누구든 같은 기준으로 자산을 올릴 수 있고, AI도 같은 구조로 분류할 수 있다.
