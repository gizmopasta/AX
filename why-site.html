BXD AI Native · Archiving Site

왜 아카이빙 사이트가
필요한가

UI 시스템 구축이 아니다. 보여주는 문서도 아니다.
만든 것들이 실제로 쓰이기 위한 구조 — 그게 사이트가 필요한 이유다.

step
01
지금 상태의 문제
만든 게 있는데
아무도 모른다
로고는 SVG로 만들었다. 컬러는 JSON으로 정리했다. Foundation 파일도 있다. GitHub URL도 있다. 근데 그 URL을 아는 사람이 없으면 — 없는 것과 같다.
URL이 있어도 공유 안 하면 없는 것
버전이 뭔지 아무도 모름
BXD한테 물어봐야 알 수 있음
드라이브 어딘가에 있는 파일 — 찾을 수 없음
쓰고 있는지조차 파악 안 됨
step
02
그래서 필요한 것
보여주는 문서가 아니라
활용하는 데이터
가이드 페이지를 만드는 게 아니다. 자산을 찾을 수 있고, 꺼내 쓸 수 있고, 실제로 활용되는지 추적되는 구조가 필요하다.
지금 — 보는 문서
로고 → 가이드 페이지에 이미지로 게시
색상 → 'Blue 500' 같은 일반 명명
사례·콘텐츠 → 참고용 모음
사용 방법 → 가이드 본문 텍스트
→
전환 후 — 활용하는 데이터
로고 → URL로 실시간 호출. 항상 최신 자동 반영
색상 → 'Action-Primary'처럼 의미 기반. AI가 맥락 이해
사례·콘텐츠 → 태그 지정, 라이브러리화, 검색·추천 가능
사용 방법 → 같이 쓸 수 있는 프롬프트 함께 제공
step
03
사이트여야 하는 이유
파일이 아니라
링크로 호출된다
드라이브에 올려두는 것과 사이트로 배포하는 것은 다르다. 사이트는 링크 하나로 접근하고, AI가 URL로 직접 자산을 참조할 수 있다.
링크 하나 — 권한 요청 없이 바로 접근
AI가 URL로 자산 직접 참조
업데이트하면 링크는 그대로, 내용만 바뀜
Figma에서 작업하다 바로 열 수 있음
추적 코드로 실제 활용 데이터 수집
step
04
만드는 사람 / 쓰는 사람
일방적인 가이드가 아니라
모두의 관점
아카이브가 풀어주는 두 사람의 문제. 일하면서 기준이 자동으로 적용되는 구조여야 의미가 있다.
만드는 사람 — BXD
케이스가 쌓여야 기준을 추출할 수 있다
근거가 있어야 규칙을 만들 수 있다
히스토리 없이는 가이드 업데이트 불가
수시로 업데이트하고 바로 게시
아카이브 = 더 나은 경험을 위한 과정
쓰는 사람 — 조직 전체
레퍼런스 찾는 시간이 줄어든다
유사 케이스에서 판단 기준을 빌릴 수 있다
승인 과정이 줄고 자율성이 올라간다
범위·제한 기준이 있어 스스로 판단 가능
아카이브 = 실전 판단의 참고서
태그와 범위·제한 기준이 핵심. 자산마다 용도·채널·맥락 태그와 "어디까지 써도 되는지"가 붙어 있어야 쓰는 사람이 BXD 없이 스스로 판단할 수 있다. 기준이 없으면 물어보는 비용이 계속 발생한다.
step
05
Figma Site여야 하는 이유
개발자 없이
BXD가 직접 만들고 게시한다
사이트 구축에 시간·인력 투자가 어려운 상황. 그렇다고 Figma 없이 일할 수도 없다. AI 호출이 가능한 툴이어야 앞으로 대응이 가능하다.
조건 일반 사이트 구축 Figma Site
개발자 단계 필요 생략 — 게시·업데이트 누르면 끝
구축 부담 큼 — 시간·비용 투자 없음 — Figma 쓰듯이
쓰는 사람의 기여 업로드·요청 거쳐야 함 바로 Figma 붙여줄 수 있음
AI 호출 여부 별도 연동 필요 가능 — URL 기반 자산 호출
업데이트 흐름 배포·릴리즈 사이클 실시간 — Figma Make 연동
활용 추적 별도 분석 툴 필요 추적 코드 심어서 데이터 수집 가능
결론
쌓아온 자산이 잠들지 않게 하는 구조.
만들고 끝이 아니라 계속 쓰이는 인프라.
로고를 SVG로 만들고, 컬러를 JSON으로 정리하고, Foundation 파일을 GitHub에 올렸다.
이 모든 것이 사이트 없이는 BXD만 아는 URL로 남는다.
아카이빙 사이트는 그 URL들이 실제로 찾아지고, 쓰이고, 쌓이는 구조다.
